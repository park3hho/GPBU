# 2주차 1교시: 소프트웨어 개발 프로세스 모형 (1)
## C3. 소프트웨어 생명주기
**정의**: 소프트웨어 자체를 하나의 생명체로 간주하고 그의 탄생부터 사망까지의 변화과정

- 정보 시스템을 개발하는 절차, 개발 단계의 반복 현상
  - SDLC(System Development Life Cycle)
- 단계
  - 타당성 조사 &rarr; 요구분석 &rarr; 설계 &rarr; 개발 &rarr; 테스트 &rarr; 유지보수 &rarr; 폐기

### (2)**역할**
- 프로젝트 비용 산정과 개발 계획을 수립할 수 있는 기본골격이 됨
- 일정 계획, 예산, 개발 요원, 자원들을 산정-분배하는데 좋은 도구가 됨
- 소프트웨어 개발의 각 단계를 뚜렷하게 구분할 수 있어 개발 진행 상황을 명확히 파악

## D4. 소프트웨어 개발 프로세스 모형
### (1) 폭포수 모형(waterfall model, 선형 순차 모형)
**정의**: 소프트웨어 개발 시 프로세스에 체계적인 원리를 도입하는 첫 방법론
- 적용사례가 많고, 가장 오래됐으며 널리 사용되는 방법
- 단계별 정의, 산출물이 명확하고 각 단계가 끝난 후 결과물이 명확히 나와야함.

ㄱ. 타당성 검토 단계  
ㄴ. 계획 단계  
ㄷ. 요구 분석 단계  
ㄹ. 설계 단계  
ㅁ. 디버깅  
ㅂ. 구현 단계  
ㅅ. 시험 단계  
    - 통합 시험 &rarr; 시스템 시험 &rarr; 인수 시험  
ㅇ. 운용 및 유지보수 단계  

### (2) 프로토타이핑 모형(Prototyping model, 원형 모형)
**정의**: 개발자가 구축할 소프트웨어의 모델을 사전에 만드는 공정으로 사용자의 소프트웨어 요구사항을 미리 파악하기 위한 메커니즘
- 사용자 요구 분석의 어려움을 해결하기 위해 실제 개발될 소프트웨어의 견본품(프로토타입)을 만들어 의사소통의 도구로 사용

#### ㄱ. 절차
요구사항 수집 &rarr; 프로토타입 설계 &rarr; 프로토타입 개발 &rarr; 고객 평가 &rarr; 프로토타입 정제 &rarr; 요구사항 수집 ....

#### ㄴ. 특징
- 개발의 타당성 검증 시 사용
- 시범 시스템 운용 가능
- 중간 변환 비용이 적음

### (3) 나선형 모형(Spiral Model)
**정의**: 폭포수 모델의 제어와 체계적인 측면에 원형 모델의 장점을 취합하여 점진적인 성과를 보면서 위험 부담을 줄일 수 있는 방법

#### ㄱ. 진화 단계
계획 수립  
위험 분석  
공학  
평가  

#### ㄴ. 특징
- 순차적 모델의 제어와 프로토 타입 모델의 반복적 특성을 수용하고, 새로운 요소인 위험 분석을 추가한 모델
- 시스템 개발 시 생기는 위험을 최소화하여 관라하고자 하는 것이 주목적
- 계속적인 변경은 소프트웨어 구조를 훼손시킬 수 있음

### (4) V-모형
**정의**: 개발 작업과 검증 작업 사이의 관계를 명백히 드러내 놓은 폭포수 모델의 변형
- 폭포수 모형이 문서와 결과물 도출에 중점을 둔 반면, V 모형은 작업과 결과의 검증에 초점을 둠.
- 의료제어시스템이나 원전제어시스템과 같은 **높은 신뢰성**이 필요한 응용 분야에 적합함

### (5) 점증적 모형
**정의**: 시스템을 여러 번 나누어 릴리스(release)하는 방법
- 중요하고 기초적인 기능을 우선 개발하여 사용하고 나머지는 다음 번에 개발하여 확장해 나가는 방법
- 실패 위험이 적음.
- 릴리스 단위가 작아야함.

### (6) UP 모형
**정의**: 한 사이클이 끝날 때마다 테스트가 완료되어 통합 및 수행 가능한 시스템이 산출되는 점증 프로세스 유형의 객체지향 개발 모형.
- 최근 소프트웨어 개발에 꼭 필요한 특성인 반복적이고 점증적인 프로세스 흐름을 제안
- 구축, 전이, 산출 단계가 수행되는 것과 동시에 다음 소프트웨어 점증을 시작함.

#### ㄱ. 개념 정립(도입, Inception)
#### ㄴ. 전개(정련, Elaboration) 
#### ㄷ. 구축
- 남아있는 덜 중요한 부분을 구축
#### ㄹ. 전환
- 테스트 설치 다음 반복 단계를 준비
#### ㅁ. 산출
- 결점 보고서와 변경 요구를 제출하고 평가

# 2주차 2교시: 소프트웨어 개발 프로세스 모형 (2)
## E5. 신속한 소프트웨어 개발 모형
- 오늘날 비지니스 환경은 기업이 소프트웨어를 신속하게 개발하여 새로운 도전과 기회에 반응할 필요가 있음.

### (1) 개요
- 요구분석, 설계, 구현 프로세스가 동시에 진행
- 사용자 요구 문서는 시스템의 가장 중요한 특성만을 정의
- 시스템은 일련의 증분으로 개발

### (2) 애자일 기법
**정의**: 소프트웨어 요구분석, 개발, 인도를 위해 반복적으로 접근법에 의존
- 시스템 요구 사항이 개발 프로레스 동안 신속하게 변경되는 비지니스 응용 개발을 기본적으로 지원
- 점증적인 개발과 동작 가능한 소프트웨어를 고객에게 빠르게 인도하기 위해 개발

**지속적인 고객 간의 커뮤니티**     
분석 &rarr; 설계 &rarr; 개발 &rarr; Prototype &rarr; 테스트 &rarr; 분석 &rarr; (...)

#### ㄱ. 애자일 기법의 원리
| 원리           | 설명                                              |
|:-------------|:------------------------------------------------|
| 고객 참여        | 고객은 개발 프로세스 전체에 긴밀하게 참여 해야함                     |
| 점증적인 인도      | 소프트웨어는 각 증분에 포함될 요구사항을 명세화하는 고객에게 점증적으로 인도      |
| 사람은 프로세스가 아님 | 팀 구성원들은 규정된 프로세스 없이 자신들의 작업방식으로 개발해야 함          |
| 변경을 수용       | 시스템 요구사항이 변경될 것을 예상하여, 이 변경들을 수용할 수 있도록 시스템을 설계 |
| 단순성을 유지      | 개발 중인 소프트웨어와 개발 프로세스 모두 단순성에 초점을 맞춤             |

#### ㄴ. 애자일 기법의 문제점
- 개발팀들이 서로 다른 장소에서 작업하고 다른 하드웨어와 소프트웨어 시스템과의 복잡한 상호작용이 존재할 수 있는 대규모 시스템 개발에는 적합하지 않음.
- 안정성이나 보안성을 요구하는 중대한 시스템 개발에 애자일 기법은 사용하지 않는 것이 좋음
    
### (3) XP 모형
**등장 배경**: XP는 가장 널리 이용되는 애자일 기법 중 하나로써 점증적 방법을 변형한 것
- 의사소통, 단순함, 피드백, 용기, 존중 &rarr; 5가지에 기초함
  - "고객에에 최고의 가치를 가장 빨리 전달하도록 하는 경량 방법론"

**정의**: 반복적인 개발과 같은 좋은 실무 관행과 고객의 참여를 극한 수준까지 유도함.
- XP 프로세스에서 고객은 구현을 위한 스토리에 우선순위를 정하는데 밀접하게 참요
- 모든 요구사항은 일련의 업무로 직접적으로 구현되는 시나리오로 표현
- 프로그래머들은 짝을 이루어 작업하고 코드를 작성하기 전에 각 업무에 대한 시험을 준비

#### ㄱ. 12가지 실천
|        |       |
|:---------|:---------|
| 계획 세우기   | 페이 프로그래밍 |
| 작은 릴리스   | 코드를 공유   |
| 메타포      | 지속적인 통합  |
| 단순한 설계   | 40시간 작업  |
| 지속적인 테스트 | 현장 고객    |
| 리팩토링     | 코딩 표준    |

### (3) 스크럼
- 30일마다 동작 가능한 제품을 제공하는 스프린트(Sprint)를 중심으로 개발함
- 개발 주기마다 적용할 기능이나 개선에 대한 목록을 제공
- 항상 팀 단위 미팅이며, 15분 정도 회의를 가짐

| 용어      | 설명                                                                                                        |
|:--------|:----------------------------------------------------------------------------------------------------------|
| 개발팀     | 페이 프로그래밍                                                                                                  |
| 제품 백로그  | 스크럼 팀이 해야할 Todo List                                                                                      |
| 제품 소유권자 | 개발에 필요한 제품의 특징이나 요구사항을 찾아서 우선순위를 정하고 프로젝트가 지속적으로 중요한 비지니스 요구를 만족할 수 있도록 계속해서 제품 백로그를 검토하는 역할을 맡은 개인 혹은 그룹 |
| 스크럼     | 매일 이루어지는 스크럼 팀 회의                                                                                         |
| 스크럼 마스터 | 스크럼 프로세스를 따르고, 팀이 스크럼을 효과적으로 활용할 수 있도록 보장하는 역할을 맡음                                                        |
| 스프린트    | 개발에서 이루어지는 반복을 의미하며, 스프린트는 주로 2~4주 길이로 구성                                                                 |

### (4) RAD, Rapid Apllication Development
- 매우 짧은 개발 주기동안 소프트웨어를 개발하기 위한 순차 모델의 응용 형태
- 컴포넌트 기반으로 개발

#### 접근 과정
- 비지니스 모델링
- 데이터 모델링
- 프로세스 모델링
- 애플리케이션 생성
- 테스팅과 인수인계

## F6. 기타 모형
### (1) 적응적 소프트웨어 개발, ASD
- 복잡한 소프트웨어와 시스템을 개발하기 위한 기법으로 제안
- ASD의 철학적 기반은 사람들 사이의 협동과 팀의 자체 조직화에 초점을 두고 있음
- 세가지 단계의 ASD '생명주기(life cycle)'를 정의
추측 - 협동 - 학습

### (2) 동적 시스템 개발 메소드, DSDM
- 통제된 프로젝트 환경에서 점증적 프로토타이핑을 통해 빠듯한 시간 제약조건 내에 시스템을 개발하고 유지보수하기 위한 프레임워크를 제공
- 애자일 소프트웨어 개발 방식이다.

### (3) 기능 주도 개발, FDD
- 여섯 단계로 구성, 처음 세 단계는 한번만, 나중 세 단계는 반복
- 기능 단위의 설계, 구축, 설치함.

### (4) 린 소프트웨어 개발, LSD
- 소프트웨어 공학에 린 제조업의 원리를 적용

### (5) AUP, Agile UP
- 도입, 정련, 구축, 전이를 채택
- 팀이 소프트웨어 프로젝트를 위해 전체적인 프로세스 흐름을 그려보는 것을 가능하게 해 주는 순차적 오버레이 제공
