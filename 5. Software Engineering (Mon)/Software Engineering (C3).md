# 3주차 1교시: 프로젝트 범위와 비용계획

## G7. 프로젝트 계획 및 문제 정의
### (1) 프로젝트 계획
**정의**: 누가 언제 무엇을 어떻게 할 것인가를 사전에 결정하는 작업

특징
- 프로젝트 계혹은 관리자가 자원, 비용, 일정을 합리적을 측정할 수 있는 체계를 제공
- 그 목적은 합리적 측정을 위한 정보발견 과정을 통해 달성

#### ㄱ. 프로젝트 핵심 관리 대상
- 3P 
  - People, Process, Problem
- 4P
  - People, Process, Product, Project

### (2) 문제 정의
**정의**: 개발의 첫 단계는 무엇을 개발할 것인가를 명확히 정의하고 개발을 계획하는 일

목표의 설정
- 사용자의 업무 현황을 조사-분석하고 현재 정보처리 문제점과 제약사항을 파악.

#### ㄱ. 타당성 분석
- 경제적 타당성: 투자 효율성, 개발 비용이나 경비 절감 (개발 우선순위 결정)  
- 기술적 타당성: 사용자가 요구하는 기능과 성능의 제공 가능성 타진  
- 법적 타당성: 지적 소유권, 프로그램 보호법

#### ㄴ. 소프트웨어 범뮈
- 기능, 성능 제한 조건, 개발 인원, 일정 계획

## H8. 프로젝트 범위 관리 프로세스
- **요구사항 수집**: 프로젝트의 목표를 이루기 위해 이해관계자의 요구사항을 정의하고 문서화하는 활동  
- **범위 정의**: 프로젝트의 범위를 기술하는 활동  
- **작업 분류 체계 작성**: WBS, 프로젝트 팀에서 프로젝트의 목표를 달성하기 위해 수행해야 할 작업을 **산출물 중심**으로 **계층적**으로 작성한 구조  
- **범위 검증**: 프로젝트 범위 관련 산출물에 대하여 고객이나 스폰서 등 이해관계자가 검토하는 활동  
- **범위 통제**: 프로젝트 범위에 대한 기준선 문서의 변경이 발생할 때 범위의 상태를 감시하고 변경을 통제하는 활동  

## I9. 비용 추정 및 비용 산정 방법
### (1) 비용 추정
**정의**: 얼마나 노력이 들어가느냐, 결국 개발 비용을 추정하는 것

- 소프트웨어 개발 비용 예측
  - 정확한 비뵹 예측은 매우 어려움
  - 알려지지 않은 요소가 산재
    - 에러 발생, 성능 저하, 엔지니어의 이직 등
  - 원가의 계산이 어려움
    - 인건비 &rarr; 이것 자체의 추정이 어려움
  - 경험적 데이터가 필요함

#### ㄱ. 예산
- 인건비
  - MM를 기초
- 경비
  - 여비, 인쇄비 재료비, 회의비 등등
- 간접경비
  - 오버헤드

#### ㄴ. 비용에 영향을 주는 요소
제품의 크기
- 제품의 크기가 커짐에 따라 비용은 기하급수로 늘어남

제품의 복잡도
- 응용 : 개발지원 : 시스템 = 1 : 3 : 9

프로그래머 자질
- 코딩, 디버깅 능력차
- 프로그래밍 언어, 응용 친숙도

요구되는 신뢰수준
- MTBF (Mean Time Between Failure)

기술 수준
- 개발 장비, 도구, 조직능력, 관리, 방법론 숙달

### (2) 프로젝트 비용을 예측하는 방법
#### ㄱ. 상향식
**정의**: 작업에 소요되는 기간을 구하고, 여기에 투입되어야 할 인력과 투입 인력의 참여도를 곱하여 최종 인건비를 계산  
- 장점: 작업에 드는 노력을 일일히 계획할 수 있음
- 단점: 개인의 주관에 의한 추정됨

#### ㄴ. 하향식
**정의**: 프로그램의 규모를 예측하고 과거 경험을 바탕으로 예측한 규모에 대한 소요인력과 기간을 추정
- 프로그램의 규모
- LOC
- 기능 점수

### (3) COCOMO
> Bohem이 제안한 원시 프로그램의 규모에 의한 예측 비용 모형
- 프로그램의 LOC를 추정
- 이에 따른 소요 MM을 예측

제품의 복잡도에 따른 분류
- Organic (유기형, 조직형)
  - 50KDSI 이하
  - 일반  응용 업무
- Semidetaached 프로젝트 (반 내장형)
  - 300KDSI 이하
  - 트랜잭션 처리 시스템, OS, DBMS 등
- embedded 프로젝트 (내장형)
  - 300KDSI
  - 하드웨어가 포함된 실시간 시스템, 미사일유도, 신호기 제어 시스템 등

#### ㄱ. 용어
KDSI: Kilo Delivered Source Instruction
MM: Man-Month
TDEV: Total Development Time

### (4) COCOMO II
- 소프트웨어 개발 프로젝트가 진행된 저오에 따라 세가지의 다른 모델을 제시
  - 각 단계에 따라 다른 계산법을 적용

1단계: 프로토타입  
2단계: 초기 설계   
3단계: 구조 설계  

### (5) 기능점수
- 정확한 라인 수는 예측 불가능
  - COCOMO 적용 불가
- 입력, 출력, 질의, 파일, 인터페이스의 개수로 소프트웨어의 규모를 나타냄
- 각 기능에 가중값이 기본이고, 복잡도에 따라 세분화함

- 총 라인 수 = FP x 원하는 언어의 1점당 LOC
- 개발 노력 = 총 라인 수 / 생산성(LOC/MM)

# 3주차 2교시: 일정 계회과 위험 분석
## J10. 프로젝트 일정
개발 프로젝트의 프로세스를 이루는 작은 작업을 파악하고 순서와 일정을 정하는 작업
- 프로젝트 개발 기간의 지연을 방지하고 프로젝트가 계획대로 진행되도록 일정을 관리하는 것
- 소프트웨어 생명주기에 기초, 개발단계가 일정 계획에 준해 잘 진행되고 있는지 감시-통제하는 것

### (1) 프로젝트 일정
작업 순서
1. 작업 분해
- WBS 작성 (Work Breakdown Structure)
2. CPM 네트워크 작성
3. 최소 소요 기간 계산
4. 소요 MM, 기간 산정하여 CPM 수정
5. 간트 차트로 그려 일정을 구체화 함.

### (2) WBS, 업무 분류식 구조
WBS, 도표 내에 있는 각 관리 단위의 성분을 밝히고 각 성분에 비용을 할당하고 이들을 합산하여 산정하는 기법
- 개발 프로젝트의 프로세스를 이루는 작은 작업을 파악하고 순서와 일정을 정하는 작업
- 큰 단위의 일을 관리하기 쉬운 작은 일로 나누기 위한 도구
- 공정들 간의 순서는 나타내지 않음

### (3) PERT
Program Evaluation and Review Technique, 프로그램 평가 및 검토 기술
- 낙관치, 기대치, 비관치 등 베타 분포를 가정하여 예측치를 계산
- 신규사업(미경험사업) 및 비반복 사업에 주로 사용

### (4) CPM
Critical Path Method, CPM 네트워크를 사용하여, 프로젝트 완성에 필요한 작업을 나열하고 작업에 필요한 소요기간을 예측하는데 사용
- CPM 네트워크는 방향성 그래프

#### ㄱ. 그리는 예
- 노드는 소작업을 표시
  - 원형 노드: 소작업의 이름과 소요 기간을 기재
  - 박스 노드: 프로젝트 중간 점검을 의미하며, 예상 종료일을 기재
- 에지는 작업 사이의 선후 관계를 표시

장점:
- 관리자의 일정 계획 수립에 도움
- 병행 작업 계획 가능

#### ㄴ. Critical Path
임계 경로: 프로젝트를 마치는데 있어서 가장 기간을 많이 차지하는 경로
- 임계 경로 내의 작업이 늦어지면 프로젝트 전체 일정이 늦어짐

#### ㄷ. 간트 차트
WBS 또는 CPM 네트워크를 기초로 프로젝트 진행 일정표 작성.
- 막대 길이는 작업 소요되는 기간을 표시함

- 막대의 흰 부분
  - 작업 수행에 소요되는 기간
- 막대의 회색 부분
  - 작업 수행 기간의 최대 연장 기간
- SLACK TIME
  - 다음 작업에 영향을 주지 않고 작업을 연장, 수행할 수 있는 기간


## K11. 조직 계획
### (1) 조직의 구성
- 소프트웨어 개발 생산성에 큰 영향을 줌

프로젝트의 구조
- 프로젝트별 조직: 프로젝트 시작에서 개발 완료까지 전담 팀 구성
- 기능별 조직: 기능별로 인원 조직
- 매트릭스 조직: 필요에 따라 요원을 차출, 끝나면 원래 소속으로 복귀

조직 구성 노하우
- 경험이 많은 사람과 적은 사람을 적당히 혼합하여 노하우 전수
- 이직의 최소화가 대규모 프로젝트 성패를 좌우함
- 프로젝트를 너무 잘게 나누면 팀(모듈)간의 의사소통 문제가 발발

### (2) 개발 조직 방법
#### ㄱ. 책임 프로그래머 팀 조직
- 의사 결정권이 리더에게 집중
- 계층적 팀 구조 

- 외과 수술 팀 구성에서 따옴
- 책임 프로그래머
  - 제품설계, 주요 부분 코딩, 중요한 기술적 결정, 작업의 지시
- 프로그램 사서
  - 프로그램 리스트 관리, 설계 문서 및 테스트 계획 관리
- 보조 프로그래머
  - 기술적 문제에 대하여 논의, 부분적 분석/설계/구현을 담당함

특징: 
- 빠른 의사결정
- 소규모 프로젝트에 적합
- 초보 프로그래머 훈련에 적합
단점:
- 책임 프로그래머 과부화 위험
- 보조 프로그래머 역할 모호

#### ㄴ. 에고레스 팀 조직
- 민주주의식 의사 결정
  - 서로 협동하여 수행하는 비이기적인 팀
  - 자신이 있는 일을 알아서 수행
  - 구성원이 동등한 책임과 권한
- 다각도의 의사 교환 경로 제공

특징:
- 작업 만족도 높음(이직률 낮음)
- 의사 교류 활성화
- 장기 프로젝트에 적합
단점:
- 책임이 명확하지 않음
- 대규모에 적합하지 않음(의사 결정 지연)

#### ㄷ. 계층적 팀 조직
- 초보자와 경험자를 분리
- 프로젝트 관리자와 고급 프로그래머에게 지휘 권한이 주어짐
- 의사 교환은 초보 엔지니어나 중간 관리층으로 분산
- 의사 전달 경로가 김

## L12. 위험 분석

**특징**: 프로젝트에 내재된 위험 요소를 인식하고 그 영향을 분석하여 이를 관리하는 활동

#### ㄱ. 개요
- 위험이 발생되었을 때의 영향을 최소화하기 위해 필요함

#### ㄴ. 위험 요소 파악

- 비현실적 일정 및 예산
