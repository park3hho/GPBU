# 5주차 1교시: 요구사항 분석의 개념
## S19. 요구사항 분석의 개념
### (1) 요구란?
> **요구**  
> 시스템이 제공해야 할 역량(Capacity)으로 외형적으로 나타내는 기능이나 성능

**현금 인출기의 예**
- 외형적 기능: 현금 인출, 잔액 조회, 계좌 이체, 현금 서비스 등
- 기능 및 성능: 시간당 처리 능력, 반응시간, 제약조건

### (2) 요구 분석
> **요구분석**  
> 사용자의 뜻을 이해하고 사용자 요구나 주어진 문제를 정확히 분석, 이해하는 과정
- 구현될 시스템의 기능이나 목표, 제약사항 등을 정확히 파악
- 사용자로부터 시스템 개발에 필요한 정보를 얻어내는 단계
- 사용자가 처음으로 참여하는 단계
- 무엇을 개발할 것인가를 결정하는 단계

### (3) **소프트웨어 시스템의 세가지 관점(모델)**
#### ㄱ. 기능 관점
- 주어진 입력에 대하여 어떤 결고가 나오는가를 보여주는 관점
- 표현 방법: 자료흐름도, 유스케이스 다이어그램

#### ㄴ. 동적 관점
- 시간의 변화에 따른 시스템의 **동작과 제어**에 초점을 맞추어 시스템의 **상태와 상태를 변하게 만드는 원인**들을 묘사하는 방법
- 표현 방법: 상태 전이도, 사건 추적도

#### ㄷ. 정보 관점
- 시스템에 사용되는 정보 객체를 찾아내고 이들 객체의 특성, 객체 사이의 관계와 연관성을 규명
- 표현 방법: ERD, Class Diagram

### (4) 요구사항 분석 모델
- 배경 모델
  - 아키텍처 모델
- 행위 모델
  - DFD 모델
  - 상태 기계 모델
- 데이터 모델
  - 객체-관계 속성 모델
- 객체 모델
  - 클래스 다이어그램

**분석 프로세스 시스템 모델**
- 데이터 흐름 모델
  - 데이터가 시스템에서 어떻게 단계별로 처리되는가를 보여줌
- 결합 모델
  - 시스템에 있는 개체가 다른 개체와 어떻게 결합되는지를 보여줌
- 아키텍처 모델
  - 시스템을 구성하는 중요한 서브시스템을 보여줌
- 분류 모델
  - 개체가 어떻게 공통 특성을 가지고 있는지를 보여줌
- 자극 반응 모델
  - 시스템이 내부 이벤트 및 외부 이벤트와 어떻게 반응하는지를 보여줌.

## T20. 요구공학
### (1) 요구공학
**정의**: 사용자 요구사항이 주의깊게 인식하여 문서화 하는것 (요구 명세서)
- 요구사항은 제공될 기능 뿐 아니라 성능, 신뢰성, 사용자 문서, 교육 등의 요구사항도 포함.

### (2) 요구공학 프로세스
요구공학 = 요구사항 개발 + 요구사항 관리

#### ㄱ. 타당성 조사
- 타당성 조사의 자료
  - 기본적인 비지니스 요구사항, 시스템의 개요, 어떻게 시스템이 비지니스 프로세스를 지원하는지에 관한 정보

#### ㄴ. 요구사항 추출 및 분석
- 요구사항 추출 및 분석은 다양한 부류의 사람이 참여
- 요구사항 발견 &rarr; 요구사항 분류와 구성 &rarr; 요구사항 우선순위 설정과 협상 &rarr; 요구사항 문서화

#### ㄷ. 요구사항 명세
- 요구사항을 상세하게 정의하는 단계
- 시스템 요구사항을 상세하고 정확하게 설명하는 것이 고객과 개발자 간의 계약의 기초가 됨

#### ㄹ. 요구사항 검증
- 요구사항이 실제 고객이 원하는 바를 정의헀는가를 검증하는 것

### (3) 요구사항 분석의 5단계
- 요구 추출
- 도메인분석
- 모델링 : 도메인 분석을 통해 찾아낸 중요한 `개념 특성 관계`에 대한 개념화
- 프로토타이핑과 시험
- 문서화 검토 : 요구 분석서 작성 및 검토

### (4) 요구의 다양성
- 기능 요구
  - 기능
  - 자료구조
  - 인터페이스
  - 사용자 관련 
- 비기능
  - 자원 관련 질문
  - 성능 관련 질문
  - 보안 관련 질문
  - 품질

### (5) 요구 추출
사용자가 무엇을 원하는지 결정을 내리는 작업

### (6) 요구 추출을 위한 의사소통기술
- 인터뷰
- 설문조사
- 브레인스토밍
- 워크샵
- 프로토타이핑
- 사용자 스토리
- 스토리보드

# 5주차 2교시: 구조적 분석과 요구사항 명세
## U21. 구조적 분석 기법
> **구조적 분석 정의**: 사용자의 요구사항을 파악하기 위해 시스템을 구성하는 프로세스들 사이의 데이터 흐름을 도표로 표현

- **프로세스 중심**의 분석 기법
- DFD 이용

**특징**
- 도형 중심의 표현
- 하향식 분해 원칙을 이용, 세분화
- 사용자의 업무 요구사항을 도형화
- 사용자와 분석자 간의 의사소통 매체로 이용
- 실 시스템 모형의 추출
  - 추상화 단계에서 점진적으로 구체화 단계로 변환

**구조적 작업의 순서**  
(1) 배경도 작성: 기본 시스템 모형의 추상화  
(2) 상위 자료흐름도 작성: 프로세스의 세분화  
(3) 하위 자료흐름도 작성: 세분화된 프로세스들의 구체화
(4) 자료 사전 작성: 자료 흐름도의 데이터 정의
(5) 소단위 명세서 작성: 최하위 프로세스 기능 명세

### (1) 배경도
특징
- 시스템과 외부환경의 관계를 표시하기 위해 작성함
- 시스템 환경분석 단계로서 시스템은 블랙박스화됨
- 배경도는 단 하나의 원으로 구성되며 DFD 레벨0을 의미함
- UML의 Use Case Diagram에 해당함.

### (2) 자료흐름도, DFD
**정의:** 자료가 소프트웨어 내의 절차에 따라 흐르면서 변화되는 과정을 도형화한 기법

- 하나의 절차가 진행될 때마다 새로운 레벨을 형성하여 자세하게 표현
- 임의의 추상화 수준에서 시스템이나 소프트웨어를 표현하기 위해 존재
- 제어(조건, 루프)에 대하여 명확하게 표기함이 없는 자료흐름을 표시

#### ㄱ. 처리
- 입력 데이터의 흐름을 출력 데이터의 흐름으로 변환
- 버블로 표현하고, 버블 내의 처리이름 표현
- 프로세스 이름(정보 변환자)은 처리 내용 또는 처리를 수행하는 행위자를 의미함.
- 번호를 붙이고, 소단위 명세서 작성

#### ㄴ. 데이터 흐름
- 이동중인 데이터 그룹을 나타냄
- 데이터 이동 방향을 화살표로 표시
- 화살표 위에 데이터 그룹의 이름을 첨가
- 데이터 저장소에 연결된 데이터 흐름은 데이터 그룹의 입출력 표시

#### ㄷ. 단말
- 시스템의 외부 엔티티로 프로세스에 의해 변환될 정보가 발생되는 또는 SW에 의해 변환된 정보를 수신하게 되는 시스템 구성요소를 표현

### (3) 자료사전
> 자료사전: 자료흐름의 대상이 되는 자료의 내용을 구체적으로 명세화

- 자료 흐름드 상의 모든 자료요소들을 수학적으로 정의함

### (4) 소단위 명세서
- **구조적 영어**: 명령문의 나열
- **의사 결정표**: 프로세스가 조건에 따라 상이한 처리를 수행할 경우 적합한 표현 방법
- NS 도표
- 전후조사

## V22. 자료구조지향 분석 기법

## W23. 요구사항 명세화
### (1) 요구사항 명세화
> 요구되는 기능의 명세화는 구현과 독립

- 소프트웨어, 정보처리시스템의 전체영역, 이용환경을 전반적으로 명세화
- 부분적 기술이 가능해야 하며, 확장 가능
