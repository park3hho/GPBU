# 1주차 1교시
## A1. 데이터베이스 발전과정 및 정의
### (1) 데이터베이스의 발전 과정
ㄱ. 1960년대 이전
- 파일 시스템의 구조
	- 종속성의 문제  

ㄴ. 1960년대: 계층형 DBMS 상용화  
- 계층형 DBMS (Hierarchial DBMS)
	- 트리 구조, 부모 자식 관계

ㄷ. 1970년대: 네트워크형 DBMS 개발
- 네트워크형 DBMS(Network DBMS)
	- 그래프 구조, 사이클 허용

ㄹ. 1980년대: 관계형 DBMS 출시
- 관계형 DBMS 
	- 테이블 구조로서 데이터베이스의 대중화 시작

ㅁ. 1990년대: 객체지향 DBMS 개발
- 객체지향 DBMS
	- 객체지향 프로그래밍 개념을 데이터베이스에 적용

**ㅂ. 1990년 중반: 객체관계 DBMS 개발**
- 객체관계 DBMS
	- 관계형 데이터베이스와 객체지향 데이터베이스 통합

**ㅅ. 1990년대 후반**
- 월드 와이드 웹의 폭발적인 성장
- 데이터베이스 시스템은 247 가용성을 갖춤
- 빅 데이터, 데이터마이닝 중요성 부각

### (2) 데이터베이스의 정의 
**정의**: 어느 한 조직의 여러 응용 시스템들이 기업의 의사결정에 활용되며 곡용할 수 있도록 통합, 저장된 운영 데이터의 집합.

통합된 데이터(integrated data): 
- 원칙적으로 데이터베이스에서는 똑같은 데이터가 중복되지 않았음을 의미하며 실제로 효율성 때문에 일부 데이터의 중복 허용
	- 최소의 중복, 통제된 중복

저장된 데이터(stored data):
- 책상서랍이나 캐비닛에 들어 있는 데이터가 아니라 컴퓨터가 접근할 수 있는 저장매체(자기테이프, 디스크)에 저장된 데이터 집합

운영 데이터:
- 어떤 조직의 고유 기능을 수행하기 위해 반드시 유지되야할 데이터

공용데이터:
- 어느 하나의 응용 프로그램이나 응용 시스템을 위한 데이터가 아니라 그 조직에 있는 여러 응용 시스템들이 공동으로 소유하고 유지하며, 이용하는 공용 데이터

### (3) 데이터베이스의 일반성
실시간 접근  
계속적인 변화  
동시공유(concurrent sharing) -> 병행제어 기법  
내용에 의한 참조(content reference)

## B2. 데이터베이스의 특징 3요소 및 구성요소
### (1) 자료 추상 
- 파일 시스템과 자료의 복잡한 내용을 감추어 복잡한 자료를 쉽게 사용
	- 자료의 핵심 내용을 추출, 사물들에서 공통의 성질을 추출하는 것
	- 추상화 = 개념화, 일반화

### (2) 자료 독립
- 자료구조를 프로그램으로 부터 독립
	- 프로그램을 변경하지 않고 자료구조 변경 가능

### (3) 자기정의
- 데이터베이스가 스스로의 내용을 정의하고 저장
	- 자료의 구성과 내용을 DB가 스스로 기억하고 관리하는 기능

## C3. 데이터베이스의 구성 요소
### (1) 개체 entity
- 데이터베이스가 표현하려고 하는 유형
	- 무형의 정보 대상
	- 서로 구별될 수 있는 것
- 컴퓨터가 취급하는 파일 구성 측면에서의 레코드에 해당
- 단독으로 존재 가능
- 하나 이상의 속성으로 구성

### (2) 속성
- 개체의 특성이나 상태를 기술하는 것
- 데이터의 가장 작은 논리적 단위
- 단독으로 존재 불가능
- 파일 구조에서는 데이터 항목 또는 필드
- 도메인이란
	- 하나의 속성이 취할 수 있는 같은 타입의 모든 값의 집합

### (3) 관계
- 개체 집합과 개체 집합 간에는 여러 가지 유형의 관계가 존재
- 속성 관계 (attribute realtionship)
	- 개체 내의 관계
	- 속서오가 속성 사이의 관계
- 개체 관계 (entity relationship)
	- 개체 간의 관계
	ex) 학생과 교수의 지도 관계

개체타입 (이라고도함)
- 스킴 
- 스키마
- 릴레이션의 내포
개체의 집합 (이라고도함)
- 릴레이션 인스턴스
- 튜플의 집합
- 개체 어커런스의 집합
- 릴레이션 외연

## D4. 데이터베이스의 구조
### (1) 논리적 구조
- 사용자가 생각하는 데이터의 논리적 표현
- 논리적 구조에서 취급하는 데이터가 논리적 레코드

### (2) 물리적 구조
- 디스크나 테이프와 같은 저장장치 위에 물리적으로 저장되어 있는 데이터의 실제 구조
- 저장장치의 입장에서 본 데이터베이스의 구조
- 물리적 구조에서 취급하는 데이터가 **저장 레코드**

### (3) 대응관계
- 하나의 데이터베이스를 표현하는 논리적 구조와 물리적 구조는 당연히 서로 대응관계를 가짐으로써 동등성을 유지

## E4. 파일시스템의 특징 및 DBMS의 장단점
### (1) 파일 시스템 개요
특징: 
- 응용 플그래머는 물리적 데이터 구조에 대해 잘 알고 있어야만 데이터에 대한 접근 방법을 응용 프로그램 속에서 효율적으로 구현이 가능
- 파일 시스템에서의 가장 큰 문제점
	- 데이터 종속성(data dependency)
	- 데이터 중복성(data redundancy) 

### (2) 파일시스템의 개요
**중복의 문제점**   

ㄱ. 일관성 결여
- 데이터 간에 불일치로 인해 데이터 일관성이 없어지며 모순성 발생 
이로 인해 동일 수준보안 불가

ㄴ. 낮은 경제성
- 갱신 비용이 추가로 발생하게 되서 그럼. 여러군데 저장하기도 해서

ㄷ. 무결성 유지 어려움
- 이러한 이유로 무결성(정확성)이 무너짐

ㄹ. 중복의 장점 
- 읽기 전용 트랜잭션의 가용성 증가
- 사이트 간의 데이텨 이동 최소화
- 사이트 고장 시 질의처리 원활

## F5. DBMS의 구성요소
### (1) 데이터베이스 관리 시스템의 정의
**정의**: 응용 프로그램과 데이터베이스의 중재자로서 모든 응용 프로그램들이 데이터베이스를 공용할 수 있도록 관리해주는 소프트웨어 시스템

**특성:**
- 파일시스템에서 야기된 종속성과 종복성 문제를 해결하기 위해 사용
- 선진 개발 방법론의 도입으로 프로그램의 단순화/ 개발 기간 단축
- 자료 독립, 표준화로 유지보수 비용 감소
- 데이터베이스에 대한 모든 접근 처리 가능

### (2) DBMS의 필수 기능
**ㄱ. 정의 기능(Definition Facility)**
- 다양한 응요프로그램과 데이터베이스가 서로 인터페이스를 할 수 있는 방법 제공
- 구조를 생성한다고 생각하면됨. (테이블 구조를 만듦)
- 데이터베이스의 구조를 정의, 저장, 레코드 형태에서 키(key)를 지정

**ㄴ. 조작 기능(Manipulation Facility)**
- 사용자와 데이터베이스 사이의 인터페이스를 위한 수단 제공
- 체계적인 연산을 지원하는 도구(언어)를 통해 구현

**ㄷ. 제어 기능(control facility)**
- 정확성과 안정성을 유지하기 위한 기능
- 무결성
- 보안, 권한, 병행제어

### (3) DBMS의 장단점
**장점**
- 데이터를 통합해서 구성함으로써 중복을 사전에 통제
- 데이터 공용 가능
- 데이터의 일관성 및 무결성 유지
- DBMS가 정당한 사용자, 허용된 데이터와 연산 등을 확인 검사함으로써 모든 데이터에 대한 철저한 보안 가능

**단점** 
- 운영비 
- 24시간 서비스, 복잡한 구조, 다중 사용자 등으로 인해 장애가 났을 때 파악하기 어려움
- 데이터베이스 시스템은 통합된 시스템이기에 시스템의 문제가 전체 시스템을 정지시킬 수 있음.


### (4)  DBMS의 구성요소
**ㄱ. DDL 컴파일러** 
- DDL로 명세된 스키마
- 즉, 메타 데이터를 처리하여 시스템 카탈로그에 저장

**ㄴ. 질의 처리기**
- 터미널을 통해 일반 사용자가 요청한 고급 질의어를 파싱하고 분석한 뒤 커파일하며 질의 개수 최적화

**ㄷ. DML 예비 컴파일러**
- 응용 프로그래머가 호스트 프로그래밍 언어로 작성된 응용 프로그램 속에 삽입되어 있는 DML 명령문들을 추출

**ㄹ. DML 컴파일러**
- DML 예비 컴파일러가 넘겨준 DML 명령어를 파싱하고 컴파일하여 목적코드를 생성

**ㅁ. 런타임 데이터베이스 처리기**
- 목적코드를 실행하며 응용프로그램이 실행ㅎ

**ㅂ. 트랜잭션 관리자** 
- 무결성 검사, 권한검사 수행
- 병행 제어함
- 회복제어

**ㅅ. 저장 데이터 관리자**
- 디스크에 저장되어 있는 사용자 데이터베이스나 시스템 카탈로그 접근을 책임짐
- 입출력 권한을 넘겨주는 역할 수행

## G6. 데이터 독립성과 3단계 데이터베이스
### (1) 데이터 독립성
**<개요>**  
**궁극적인 목표**: 응용 프로그램이 데이터에 종속되지 않는 데이터의 독립성(data independency)를 제공하는 것
- 상위 단계의 스키마 정의에 영향을 주지 않고 하위 단계의 스키마 정의를 수정할 수 있는 능력
#### ㄱ. 논리적 데이터 독립성
- 프로그램에 영향을 주지 않고 데이터베이스의 논리적 구조를 변경시킬 수 있는 능력
- 응용 프로그램과 자료구조를 독립시키는 것.

#### ㄴ. 물리적 데이터 독립성
- 응용프로그램이나 데이터베이스의 논리적 구조에 영향을 주지 안혹, DBMS가 데이터의 물리적 구조를 변경할 수 있는 것

#### ㄷ. 3단계 데이터베이스
- 스키마 정의
	- **데이터 구조**와 **제약 조건**에 관한 명세를 기술한 것으로 컴파일되어 데이터 사전(메타 데이터 형태로)에 저장
- 스키마
	- 데이터 구조를 표현하는 개체
	- 개체의 특성을 표현하는 속성
	- 이들 간 존재하는 관계에 대한 정의
	- 이들이 유지해야 될 제약 조건

**사용자**의 견해: 외부스키마  
사용자 저장장치 **사이**의 견해: 개념스키마  
**저장장치**의 견해: 내부스키마  

인스턴스란? 스키마의 반대 개념으로 어느 특정한 순간에 데이터베이스에 저장되어 있는 정보들의 모임


### (2) 3단계 데이터베이스
#### ㄱ. 외부 스키마
- 사용자를 위한 논리단계 및 자료 구조라 생각하면 좋음
- 사용자는 다양하기에 여러개가 존재
- 한 논리적인 부분을 보여주기에 서브 스키마라고도 한다.

#### ㄴ. 개념 스키마
- 범 기관적 입장으로 한개만 존재
- 개체, 관계, 제약조건, 접근권한, 보안정책, 무결성 규칙에 관한 명세 포함
- 조직 논리 단계(community logic level)
	- 단순히 스키마라고 불림

#### ㄷ. 내부 스키마
- 저장장치 입장에서 데이터베이스 전체가 저장되는 방법을 명세한 것으로 한 개만 존재
- 실제로 저장될 내부 레코드의 형식, 인덱스의 유무, 저장 데이터 항목의 표현 방법, 그리그 내부 레코드의 물리적 순서 등에 관한 명세 포함
- 내부 스키마는 실제로 물리적(하위단계: 운영체제) 단계보다 한칸 위에 있음


## H7. 데이터 사전과 데이터 언어, 사용자 분류
### (1) 데이터베이스 시스템의 구조
#### ㄱ. 개요
- 데이터를 DB로 저장하고 관리해서 필요한 정보를 생성하는 컴퓨터 중심의 시스템

#### ㄴ. 데이터베이스 시스템의 구성요소
- Database
- DBMS
- DataLanguage
- DBA

#### ㄷ. 데이터 사전 
**정의**: 데이터베이스에 포함되는 모든 데이터 개체들에 대한 정의-명세에 관한 정보를 유지관리하고 있는 시스템
- 여러 스키마와 이들 속에 포함된 사사들에 관한 정보도 컴파일 되어 저장, 관리
- 질의 최적화 모듈이나 보안 및 권한에 관한 모듈 정보 저장 

- 시스템 카탈로그 자체도 일반 사용자 데이터 테이블과 같이 릴레이션, 즉, 테이블로 구성되어 있어 일반 질의어로 검색 가능
- 카탈로그의 정보
 - DML로 메타데이터 갱신 불가
 - DBMS가 관리함.

#### ㄹ. 데이터 디렉터리
**정의**: 데이터 사전에 수록된 데이터를 실제 접근하는 데 필요한 정보를 관리-유지하는 시스템
- 시스템 카탈로그와 데이터 디렉터리는 모두 시스템 데이터베이스라는 점에서는 같음. 다만 데이터 디렉터리는 **시스템**만 접근할 수 있음

명령어
- 생셩 삭제 변경

- 절차적 데이터 조작어: 정확한 절차 방식이 기술되어 있음, 저급 언어

## I8. 병렬 데이터베이스 시스템 
- 고속의 내부 연결 네트워크로 연결되어 여러 처리기와 여러 디스크로 구성
	- 소수의 강력한 처리기로 구성
	- 대량 별렬 또는 작은 단위의 컴퓨터는 수천개의 보다 작은 컴퓨터를 사용
### (1) 공유 메모리 구조 (메모리를 통해 데이터 접근함)
- 처리기와 디스크는 일반적으로 버스 또는 내부 연결 네트워크를 통해 공통 메모리에 액세스
- 공유 메모리 내의 데이터는 소프트웨어를 사용해 이동시킬 필요 ㅇ벗이 어떤 처리기에서도 액세스 가능
- 메모리에서 병목이 잘 생겨서 병렬성 정도가 낮은 경우에 널리 사용

### (2) 공유 디스크 구조 (얘도 메모리를 통하긴 한다만, 메모리가 많아서)
- 디스크 내부의 연결에서 병목이 생김
- 처리기 간의 통신이 느려짐
